// What blend ships in Button.d.ts — full of references:
declare const Button: ForwardRefExoticComponent<{
  buttonType?: ButtonType;        // imported from './types'
  text?: string;
  width?: string | number;       // a multi-type prop
  onClick?: (e: MouseEvent) => void;
} & Omit<ButtonHTMLAttributes<HTMLButtonElement>, "style">>;
😖 Regex tools choke here: they can't follow Omit<…> into React's files, or the ButtonType import. They drop those props.
// We don't read text — we ASK the TypeScript type-checker.
const checker = program.getTypeChecker()
const props  = checker.getPropertiesOfType(buttonProps)

// The checker hands back ONE flat, fully-resolved list:
buttonType   → enum "primary" | "secondary" | "danger" | "success"
text         → string
width        → string | number
onClick      → (MouseEvent) => void
// …PLUS everything Omit pulled in from React, already merged:
id, name, form, tabIndex, aria-label, onBlur, onFocus …  (57 more)
✨ getPropertiesOfType resolved Omit, the import, and the intersection for free. This is the whole advantage.
// extract.mjs turns each prop into a small, neutral data object (the IR):
{
  enums: [
    { name: "buttonType", members: [
      { as: "primary", ctor: "Primary" }, … ] }
  ],
  unboxed: [
    { name: "widthValue", members: [
      { ctor: "Str", type:{kind:"string"} },
      { ctor: "Num", type:{kind:"number"} } ] }
  ],
  props: [
    { name:"buttonType", type:{kind:"typeRef",to:"buttonType"}, optional:true },
    { name:"width",      type:{kind:"typeRef",to:"widthValue"}, optional:true },
    { name:"onClick",    type:{kind:"callback",arg:"ReactEvent.Mouse.t"} } …
  ]
}
The IR is generator-neutral — just facts about the types. emit.mjs reads it next.
type buttonType =
  | @as("primary") Primary
  | @as("secondary") Secondary
  | @as("danger") Danger
  | @as("success") Success

@unboxed type widthValue = Str(string) | Num(float)

@module("@juspay/blend-design-system") @react.component
external make: (
  ~buttonType: buttonType=?,
  ~text: string=?,
  ~width: widthValue=?,
  ~onClick: ReactEvent.Mouse.t => unit=?,
  ~id: string=?,
  @as("aria-label") ~ariaLabel: string=?,
) => React.element = "Button"
✅ Compiles on ReScript 12. width=Num(5.0) sends raw 5 to blend; width=Str("100%") sends "100%".

4 The 6 compiler-API steps (in extract.mjs)

📍 1 — Load the file and everything it imports 
ts.createProgram([entryFile], options)   // also pulls in ./types, ../Skeleton, react

TypeScript's real loader follows every import. Nothing skipped.

🧠 2 — Get the "brain" (type-checker) 
const checker = program.getTypeChecker()

Ask it anything about a type, it answers with the fully-resolved truth.

🔎 3 — Find the component export 
checker.getExportsOfModule(moduleSymbol)  // → Button (default / first Capitalized)
🎯 4 — Get the props type 
compType.getCallSignatures()[0].parameters[0]  // a component is "a fn you call with props"
✨ 5 — The magic line: flatten all props 
checker.getPropertiesOfType(propsType)  // Omit/intersection/imports already resolved

This is what regex can't do. One call → the complete, flat prop list.

🗺️ 6 — Resolve & map each prop 
for (const p of props) {
  const t = checker.getTypeOfSymbolAtLocation(p)  // real type
  classify(t)   // → IR → ReScript
}

6 Why @unboxed matters (the key trick)

❌ Tagged variant (wrong) 
type width = Str(string) | Num(float)
Str("100%") → {TAG:"Str",_0:"100%"}

Blend gets a wrapper object, not the value. Broken at runtime.

✅ Untagged (@unboxed) 
@unboxed type width = Str(string) | Num(float)
Str("100%") → "100%"

Wrapper erased at compile time. Blend gets the raw value. Type-safe + correct.
